iOS Native App: Effort, Challenges & Feasibility

An honest assessment of what building a native iOS application would involve, and whether it aligns with the project's hardware-first design philosophy.

Technical Challenges

  • Audio latency — iOS AudioUnit/AVAudioEngine needs careful buffer management to match the <10ms target
  • Background audio — GPS tracking + mic capture + radio streaming simultaneously requires complex background mode handling
  • Web Audio parity — Re-implementing the entire audio engine (synth, sampler, sequencer, FX chain, mixer) in Swift/ObjC
  • MIDI/hardware bridge — CoreMIDI for external controllers, but no direct GPIO/I2C access like Teensy firmware
  • Radio streaming — No FM receiver hardware; limited to internet radio via AVPlayer
  • App Store restrictions — Background GPS + mic access triggers privacy review scrutiny

Development Effort

  • Audio engine — Core Audio / AVAudioEngine graph: sequencer, synth, sampler, mixer, FX (major effort)
  • UI rebuild — SwiftUI or UIKit recreation of all panels, controls, knobs, faders, sequencer grid
  • AI integration — Claude API client, pattern generation logic (moderate port)
  • GPS/Maps — MapKit + CoreLocation for journeys, landmarks (straightforward)
  • Data persistence — CoreData or SwiftData for scenes, recordings, settings
  • Testing — XCTest + UI testing for all audio/GPS/control flows
  • Apple Developer account — $99/year, review process, provisioning

Why iOS Conflicts with the Vision

  • No tactile controls — Touch-only interaction defeats the purpose of a physical instrument with knobs, faders, and pads you can feel
  • Screen dependency — Musicians need eyes-free operation; a phone screen demands visual attention
  • No piezo/contact mic — iOS has no analog sensor inputs for vibration capture
  • No FM radio — iPhone has no FM receiver; a core feature (local radio capture) would be internet-only
  • Battery competition — Audio + GPS + radio drains phone battery needed for other tasks
  • Latency risk — iOS audio pipeline adds overhead vs. dedicated Teensy hardware

Better Alternatives

  • Raspberry Pi kiosk — Run the web app in Chromium fullscreen on a 5" touchscreen with GPIO for hardware controls (~$187 BOM)
  • Teensy 4.1 firmware — Purpose-built hardware with <5ms latency, direct I2C/SPI, analog inputs (already designed)
  • iOS companion app — Lightweight remote control via BLE/WiFi to the hardware device (scene recall, GPS share, monitoring)
  • PWA enhancement — Improve the existing web app as Progressive Web App: offline support, home screen install, Web MIDI for external controllers

Verdict

A native iOS app would require rebuilding the entire audio engine, UI, and interaction model from scratch — but would lose the core qualities that make Oh My Ondas a real instrument: tactile knobs you can twist without looking, faders with physical travel, pads with actual pressure response, and dedicated analog inputs (piezo, FM radio). The result would be "yet another synth app on the App Store" rather than a location-aware hardware instrument.

Recommended path: Build the Teensy/Raspberry Pi hardware device. Use the existing web app as the development/testing prototype. If mobile access is needed, create a lightweight iOS companion app that communicates with the hardware via BLE — for remote monitoring, scene management, and GPS data sharing. Not as a replacement for the instrument itself.
