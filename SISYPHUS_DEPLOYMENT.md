# Sisyphus Deployment Rules

> **What is this file?** This defines how this project is deployed on [Sisyphus](https://sisyphus.levm.eu), the hosting platform. It is read automatically by Claude Code (AI coding assistant) to enforce deployment constraints. Humans: read the rules below — they apply to you too.

This project is a **static site** deployed on Sisyphus. These rules apply when creating, editing, or reviewing code.

## Deployment Target

- **Platform:** Caddy v2 file server on Flatcar Container Linux
- **HTTPS:** Automatic (Let's Encrypt) — never handle TLS in application code
- **Monitoring:** SRE agent with health checks every 60s, auto-restart, Telegram alerts
- **Health check:** `GET /` must return HTTP 200 (Caddy serves index.html automatically)
- **What gets served:** Static files only. If the project has a build step (React, Vue, Astro, Hugo, etc.), the build output is what gets deployed.

## Hard Rules — deployment fails without these

1. `index.html` MUST exist in the served directory (repo root, or build output directory)
2. ALL asset paths MUST be relative (`./css/style.css`, `./images/logo.png`) — never absolute (`/css/style.css`)
3. NO server-side code in production — no PHP, Python, Node.js runtime. HTML, CSS, and client-side JavaScript only
4. NO secrets anywhere — no API keys, tokens, passwords in code or git history. Use environment variables for third-party service keys.
5. NO sensitive files committed: `.env`, `*.key`, `*.pem`, `*.p12`, `credentials.*`, `*secret*`
6. Multi-page sites: all internal links must use relative paths (`./about.html`, not `/about`)

## Security Rules

- NEVER hardcode API keys, tokens, passwords, or connection strings
- NEVER commit sensitive files (see rule 5 above)
- Validate and sanitize all user input (forms, query parameters, URL segments)
- Set `rel="noopener noreferrer"` on external links with `target="_blank"`
- Prefer vendoring dependencies locally over CDN scripts. **Exception:** third-party services that mandate CDN loading (Stripe.js, Snipcart, reCAPTCHA, payment providers) — use their CDN as required.

## Recommended — not enforced, but follow unless there's a reason not to

- `<meta charset="UTF-8">` and `<meta name="viewport" content="width=device-width, initial-scale=1.0">` in every HTML file
- All images have `alt` attributes (WCAG AA accessibility)
- Keep files under 10 MB each, total repo under 100 MB
- Prefer CSS custom properties over preprocessors (no Sass/Less)
- Prefer system fonts unless the project has specific typography requirements
- Mobile-first responsive design — test at 375px, 768px, 1280px

## Build Steps

Build tools are fine if the project needs them. The platform serves the build output, not the source.

- If using React, Vue, Svelte, Astro, Hugo, etc. — the build output directory must contain `index.html` and use relative paths
- Tell the infra operator the build command and output directory during onboarding
- Do NOT commit build output to git — gitignore `dist/`, `build/`, `out/`, etc.
- `node_modules/` must be gitignored

## Git Conventions

- Commit messages: imperative mood, under 72 characters (e.g., "Add contact page", "Fix navigation on mobile")
- Commit frequently — small, focused commits
- `main` is the deployment branch — push to `main` to deploy
- NEVER force-push to `main`
- NEVER commit generated files, build artifacts, or dependencies

## What NOT to Do

- Do NOT create Docker, CI/CD, or infrastructure configuration — this is a static site
- Do NOT add analytics or tracking without explicit user request
- Do NOT over-engineer — a portfolio does not need React, a blog does not need a database
- Do NOT modify this CLAUDE.md file

## Quality Assurance — Before Handover

Every project MUST include tests that validate quality before deployment.

| Category | What to test | Required |
|----------|-------------|:---:|
| **Regression tests** | All pages render, all links resolve, all forms submit | Yes |
| **Unit tests** | Individual functions, components, modules | If JS exists |
| **Security tests** | XSS, injection, secrets exposure | Yes |
| **Accessibility tests** | WCAG AA, keyboard navigation, screen readers | Yes |
| **Performance tests** | Page load < 3s, asset sizes reasonable | Yes |

Tests must be runnable with a single command, documented in README under `## Testing`.

## Pre-Push Validation

```bash
./validate.sh    # exit 0 = ready to deploy
```
